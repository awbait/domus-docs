# Фокус — Персональный дашборд

**Статус:** Проектирование
**Последнее обновление:** 2026-03-17

---

## Видение

Персональный дашборд для повседневной жизни. Висит на стене как умный экран, открывается в браузере и на телефоне. Модульная архитектура — каждый модуль независимый и может быть написан сторонним разработчиком.

**Не клон Home Assistant.** Это про людей и задачи, а не про устройства. HA управляет устройствами, Фокус — жизнью. Домовой интегрируется как модуль для голосовых команд и управления hub/spoke.

---

## Целевые сценарии использования

| Сценарий | Устройство | Условие |
|----------|-----------|---------|
| Быстрый взгляд проходя мимо | Настенный монитор (тач) | Дома, всегда включён |
| Отметить таблетку коту | Настенный монитор / телефон | Дома |
| Проверить список дел | Телефон | В дороге |
| Управление устройствами (через HA) | Настенный монитор / телефон | Дома |
| Утренняя сводка | Настенный монитор | Дома |
| Текстовая команда Домовому (чат с LLM) | Настенный монитор / телефон | Дома / в дороге |

## Принятые решения

| Вопрос                          | Решение                                          | Почему                                                                                                                             |
| ------------------------------- | ------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------- |
| Frontend-фреймворк              | **React + Vite + TypeScript**                    | Большая экосистема, проще найти компоненты для grid/drag-drop. Vite вместо Next.js — SSR не нужен для дашборда, билдится в статику |
| UI-компоненты                   | **shadcn/ui** (Radix UI + Tailwind)              | Copy-paste в проект, полный контроль над стилями. Легко кастомизировать под wall mode, акцентный цвет через CSS variables           |
| Grid-сетка                      | **react-grid-layout**                            | Стандарт для дашбордов (Grafana). Drag-drop, resize, responsive breakpoints, сохранение layout                                     |
| Backend-язык                    | **Go**                                           | Уже основной язык проекта, нетребователен к ресурсам на Orange Pi, опыт с chi/SQLite/gRPC                                          |
| Хранилище                       | **SQLite** (сейчас), миграция на Postgres позже   | Один процесс, минимум ресурсов. Repository pattern позволяет сменить БД без переписывания логики                                   |
| Репозиторий                     | **Отдельный** в `domus/focus-dashboard/`         | Независимый деплой от Домового                                                                                                     |
| Мультиплатформенность           | **PWA**                                          | Веб + мобильное (Add to Home Screen) + десктоп — одна кодовая база. Push, offline, fullscreen                                      |
| Модули сторонних разработчиков  | **Из папки** (Фаза 1), **registry-репо** (позже) | Простое начало, усложнение по мере надобности                                                                                      |
| Перезапуск при установке модуля | **Да**, перезапуск всегда                        | Go `plugin` пакет не поддерживает выгрузку, hot-reload нереалистичен. `docker compose restart` — секунды                           |

## Технический стек

### Frontend
- **Фреймворк:** React 19 + TypeScript
- **Сборка:** Vite — билдится в статику, Go раздаёт через `embed`
- **UI-компоненты:** shadcn/ui (Radix UI + Tailwind). Copy-paste в проект — полный контроль над стилями, легко адаптировать под wall mode (крупные шрифты, 48px targets)
- **Grid-сетка:** react-grid-layout — drag-drop, resize виджетов, responsive breakpoints, сохранение layout per-user
- **Стиль:** Tailwind CSS
- **Иконки:** Lucide (идёт с shadcn/ui)
- **PWA:** service worker, иконка на главный экран, push-уведомления (Web Push API), offline-кэш
- **Темы:** тёмная (настенный ночной режим) + светлая. Через CSS variables (shadcn поддерживает из коробки)
- **Кастомизация:** акцентный цвет (per-user, через CSS variable `--primary`), фоновое изображение (загрузка или URL)
- **Адаптивность:** 3 режима рендера — `wall` (1080p touch), `desktop` (браузер), `mobile` (телефон)

### Backend
- **Язык:** Go (отдельный сервис от Домового)
- **Роутер:** chi
- **API:** REST + WebSocket (realtime обновления виджетов)
- **Хранилище:** SQLite (через `modernc.org/sqlite` — без CGo)
- **Аутентификация:** OIDC (Authentik) + локальные сессии (JWT)
- **Порт:** 8080
- **Связь с Домовым:** HTTP API (для модуля управления)
- **Документация API:** Swagger UI (`/api/swagger/`). Генерация через `swaggo/swag` — аннотации в комментариях хэндлеров. Модули используют те же аннотации → их ручки автоматически попадают в общий spec

### Деплой
- Docker-контейнер (standalone или K3s pod рядом с Домовым)
- Frontend встроен в Go-бинарник через `embed`

## Модульная архитектура

### Принцип

Каждый модуль — самостоятельная единица. Ядро предоставляет инфраструктуру:
- **Auth** — аутентификация, роли, сессии
- **Layout** — grid-сетка, drag-drop, сохранение per-user layout
- **WebSocket** — realtime обновления для виджетов
- **Push-уведомления** — VAPID ключи, хранение подписок браузеров, отправка. Модули вызывают `core.SendPush(userID, title, body)`, ядро доставляет
- **Хранилище настроек** — per-user и per-module настройки в SQLite
- **Внешние ссылки** — быстрые ссылки на сервисы в сети (HA, Jellyfin, Immich и т.д.), настраиваются админом, отображаются на дашборде
- **Бэкапы** — автоматический бэкап SQLite, экспорт/импорт

### Типы модулей

| Тип           | Описание                                                   | Пример                               |
| ------------- | ---------------------------------------------------------- | ------------------------------------ |
| **frontend**  | Только виджет, без бэкенда. Работает на клиенте.           | Часы, фото-слайдшоу, заметки-стикеры |
| **fullstack** | Виджет + бэкенд (Go). Свои роуты, миграции, бизнес-логика. | Таблетница, управление Домовым       |

### Манифест модуля

```json
{
  "id": "pill-tracker",
  "name": "Таблетница",
  "type": "fullstack",
  "version": "1.0.0",
  "author": "username",
  "description": "Отслеживание приёма таблеток для людей и животных",
  "icon": "pill",

  "defaultSize": "2x2",
  "hasConfig": true,
  "permissions": ["notifications", "storage"],
  "backendRoutes": "/api/modules/pill-tracker"
}
```

Для frontend-only модуля: `"type": "frontend"`, `"backendRoutes"` отсутствует.

### Структура модуля (frontend)

```
frontend/src/modules/
  pill-tracker/
    manifest.json
    Widget.tsx          # рендер на дашборде
    ConfigPanel.tsx     # настройки модуля
    FullScreen.tsx      # полноэкранный режим (тап на виджет)
    hooks/
      usePills.ts
    api.ts              # запросы к бэкенду
```

### Структура модуля (backend Go)

```
modules/
  pill-tracker/
    handler.go          # HTTP handlers
    service.go          # бизнес-логика
    repository.go       # работа с БД
    model.go            # типы данных
    migrations/
      001_init.sql
```

### Контракт между ядром и модулем

```go
// Интерфейс который реализует каждый backend-модуль
type Module interface {
    ID() string
    RegisterRoutes(r chi.Router)
    Migrate(db *sql.DB) error
    HealthCheck(ctx context.Context) error
}
```

Хэндлеры модуля должны содержать `swaggo/swag` аннотации — тогда `swag init` автоматически включит их в общий Swagger spec.

### Жизненный цикл модуля

1. **Установка:** модуль кладётся в папку `modules/` (вручную или через registry)
2. **Активация:** включается в админке → требуется перезапуск
3. **Обновление:** заменяются файлы → требуется перезапуск. В админке отображается "Доступно обновление" (если registry)
4. **Удаление:** выключается в админке → файлы удаляются → требуется перезапуск

### Перезапуск из UI

При изменении модулей (установка/обновление/удаление) фронтенд показывает баннер "Требуется перезапуск" с кнопкой "Перезапустить сейчас". После нажатия:

1. Фронт отправляет `POST /api/admin/restart`
2. Бэкенд инициирует graceful shutdown
3. Фронт переключается в режим ожидания: лоадер + polling `GET /api/health` каждые 2 секунды
4. Когда health отвечает 200 → фронт автоматически перезагружает страницу
5. Если бэкенд не отвечает >60 секунд → показывает ошибку "Не удалось перезапустить"

> Бэкенд перезапускается через Docker API или systemd — зависит от деплоя. В Docker: контейнер сам себя рестартит через `restart: unless-stopped` policy после `os.Exit(0)`.

### Grid Layout

Дашборд — сетка. Каждый виджет занимает ячейки:
- `1x1` — маленький (часы, погода кратко)
- `2x1` — горизонтальный (следующее событие)
- `1x2` — вертикальный
- `2x2` — стандартный (таблетница, погода детально)
- `4x2` — широкий (календарь, список задач)

Пользователь перетаскивает виджеты на настенном и десктопном режиме.

### Персонализация Layout

У каждого пользователя свой сохранённый layout:
- Позиция и размер каждого виджета на сетке
- Скрытые виджеты (установлен модуль, но пользователь не хочет видеть)
- Настройки виджетов (например, какого пациента показывать в таблетнице)

Отдельный layout для каждого режима (`wall`, `desktop`, `mobile`) — экраны разные, расположение тоже.

Shared-режим настенного монитора имеет собственный layout, редактируемый админом.

## Модуль: Таблетница (MVP #1)

**Тип:** fullstack

### Задача

Коту нужно давать 6 таблеток в день. Важно:
- Видеть что ещё не дано
- Одним тапом отметить как данное
- Задать расписание (когда какая таблетка)
- Получить уведомление если пропустил время
- История — кто и когда давал (полезно если в доме несколько человек)

### Сценарии использования

1. Подхожу к монитору → вижу какие таблетки сегодня остались
2. Дал таблетку → тапаю галочку → отмечается с временем и именем
3. Не дал вовремя → пуш-уведомление на телефон
4. Смотрю историю → "14 марта 08:15 — Иван дал Рибоксин"

### Данные

```
Patient (пациент — кот, человек, кто угодно)
  - id
  - name          "Барсик"
  - type          "cat" | "human"
  - avatar        emoji или фото

Medication (лекарство)
  - id
  - patient_id
  - name          "Рибоксин"
  - dosage        "1 таблетка"
  - color         для визуального различия
  - notes         "давать с едой"

Schedule (расписание)
  - id
  - medication_id
  - time          "08:00"
  - days          ["mon", "tue", "wed", "thu", "fri", "sat", "sun"] или конкретные даты
  - active        bool

DoseLog (журнал)
  - id
  - schedule_id
  - planned_at    время по расписанию
  - given_at      фактическое время
  - given_by      user_id залогиненного пользователя
  - skipped       bool
  - notes         "кот не ел, дал позже"
```

### UI виджет (2x2)

```
┌─────────────────────────┐
│  Барсик — Сегодня       │
├─────────────────────────┤
│  [v] 08:00  Рибоксин    │
│  [v] 10:00  Метипред    │
│  [!] 14:00  Рибоксин    │  <- текущее время
│  [ ] 18:00  Метипред    │
│  [ ] 20:00  Рибоксин    │
│  [ ] 22:00  Метипред    │
├─────────────────────────┤
│  3 из 6 выполнено       │
└─────────────────────────┘
```

Тап на `[ ]` → появляется подтверждение "Дал Метипред в 14:02?" → подтверждаю → отмечается.

### API

```
GET  /api/modules/pill-tracker/today?patient=barsik
     → список доз на сегодня со статусом

POST /api/modules/pill-tracker/doses/{id}/give
     body: { given_by: "Иван", notes: "" }
     → отмечает дозу как данную

POST /api/modules/pill-tracker/doses/{id}/skip
     body: { reason: "кот не ел" }

GET  /api/modules/pill-tracker/history?patient=barsik&days=7

GET  /api/modules/pill-tracker/patients
POST /api/modules/pill-tracker/patients

GET  /api/modules/pill-tracker/medications
POST /api/modules/pill-tracker/medications

GET  /api/modules/pill-tracker/schedules
POST /api/modules/pill-tracker/schedules
PUT  /api/modules/pill-tracker/schedules/{id}
```

### Уведомления

- Web Push на телефон через service worker
- Напоминание за 5 минут до времени
- Напоминание если не отмечено через 30 минут после планового времени
- Если подключён Домовой — голосовое напоминание: "Барсику пора давать Рибоксин"

## Модуль: Управление Домовым (MVP #2)

**Тип:** fullstack

### Задача

Панель управления голосовым ассистентом Домовой: текстовые команды, история, логи, управление hub/spoke инстансами. Управление устройствами (свет, климат) — через Home Assistant (ссылка настраивается в админке через внешние ссылки ядра).

### Сценарии использования

1. Отправляю текстовую команду Домовому (вместо голосовой)
2. Вижу историю последних голосовых команд — что распознал, что ответил
3. Смотрю логи ошибок и debug-информацию
4. Вижу статус hub и spoke инстансов — online/offline, версия, uptime
5. Настраиваю spoke: подключение нового spoke к hub, конфигурация
6. Вижу доступные обновления для hub/spoke, запускаю обновление из дашборда

### Зависимости

Требует HTTP API в Домовом:
- Выполнение текстовых команд
- История команд/ответов
- Статус hub/spoke инстансов (включая текущую и latest версию)
- Управление spoke (регистрация, конфигурация)
- Запуск обновления hub/spoke

> Это задача для Домового (Go), не для дашборда. Модуль дашборда только потребляет это API.

### Данные

```
CommandLog (история команд — из Домового API)
  - id
  - timestamp
  - transcript    "включи свет в спальне"
  - intent        "light.on"
  - response      "Включаю свет в спальне"
  - success       bool

HubSpokeNode (инстанс — из Домового API)
  - id
  - role          "hub" | "spoke"
  - name          "Кухня" | "Спальня"
  - address       "192.168.1.50:8443"
  - status        "online" | "offline" | "degraded"
  - version       "0.5.2"
  - latest_version "0.6.0"  (null если актуальная)
  - uptime        duration
  - last_seen     timestamp
```

### UI виджет (2x2)

```
┌──────────────────────────┐
│  Домовой                 │
├──────────────────────────┤
│  Hub: online   v0.5.2    │
│  Spoke кухня:  v0.5.2    │
│  Spoke спальня: v0.5.1 ! │  ← есть обновление
│                          │
│  Последняя команда:      │
│  "выключи свет" → OK     │
├──────────────────────────┤
│  [Команда...]            │
└──────────────────────────┘
```

### UI полноэкранный режим

- Текстовое поле для отправки команды Домовому
- Лог команд с фильтрацией по дате и статусу
- Список hub/spoke инстансов: статус, версия, uptime, адрес
- Индикатор обновления: если `latest_version > version` → кнопка "Обновить до v0.6.0"
- Обновление всех / по одному: hub обновляется первым, потом spoke
- Подключение нового spoke: адрес, имя, авторизация
- Настройка spoke: модели, пороги, параметры

### API

```
POST /api/modules/domovoy-control/command
     body: { text: "включи свет в спальне" }
     → отправляет текстовую команду Домовому, возвращает ответ

GET  /api/modules/domovoy-control/history?limit=50
     → последние команды и ответы

GET  /api/modules/domovoy-control/nodes
     → список hub/spoke инстансов со статусом

GET  /api/modules/domovoy-control/nodes/{id}
     → детальная информация об инстансе

POST /api/modules/domovoy-control/nodes
     body: { name: "Кухня", address: "192.168.1.50:8443", role: "spoke" }
     → регистрация нового spoke

PUT  /api/modules/domovoy-control/nodes/{id}/config
     body: { ... }
     → обновление конфигурации spoke

POST /api/modules/domovoy-control/nodes/{id}/update
     → запускает обновление инстанса до latest_version

POST /api/modules/domovoy-control/nodes/update-all
     → обновляет все инстансы (hub первым, потом spoke)

```

- Быстрые заметки (sticky notes)
- Семейные объявления ("не забудь купить молоко")

### Фото
- Слайд-шоу семейных фото

## Требования к настенному режиму

- Работает 24/7, экран не гаснет (или гаснет по расписанию — ночью)
- Шрифты крупнее чем на десктопе
- Touch targets минимум 48x48px
- Информация читается с расстояния 1.5 метра
- Нет мелкого текста, нет overflow скролла на главном экране
- Жест "свайп вверх" — полноэкранный модуль
- Жест "свайп вниз" — главный экран

---

## Требования к мобильному режиму

- **PWA** — добавляется на главный экран как приложение (iOS / Android)
- Push-уведомления (таблетница, таймеры, напоминания)
- Работает вне дома (HTTPS + auth)
- Offline: кэширует последние данные, показывает без сети
- Нижняя навигация вместо боковой

## Auth

### Первый запуск (Setup Wizard)

1. Бэкенд видит пустую БД → редирект на `/setup`
2. Создание admin-аккаунта: имя + пароль
3. Опционально: подключение OIDC провайдера (Authentik / Authelia / Keycloak)
4. Redirect на дашборд

### Методы аутентификации

| Контекст          | Метод                                            |
| ----------------- | ------------------------------------------------ |
| Локальная сеть    | Без авторизации (по настройке) или PIN           |
| Внешний доступ    | OIDC через Authentik (основной) или логин/пароль |
| Настенный монитор | Auto-login (доверенное устройство) или PIN       |

### Роли и доступ

| Роль     | Возможности                                                |
| -------- | ---------------------------------------------------------- |
| `admin`  | Полный доступ: конфигурация, модули, профили, пользователи |
| `user`   | Использование дашборда: виджеты, отметка таблеток, команды |
| `viewer` | Только просмотр: дашборд, логи, история — без изменений    |

- Маппинг из OIDC claims/groups → роли
- Несколько профилей: "Иван", "Катя" — для журнала таблеток и персонализации

### Пользователи и scope данных

Каждый виджет и его данные имеют **scope**:

| Scope      | Видимость        | Примеры                                      |
| ---------- | ---------------- | -------------------------------------------- |
| `shared`   | Все пользователи | Таблетница, погода, инвентарь, Домовой, часы |
| `personal` | Только владелец  | Задачи, календарь, трекер привычек, финансы  |

Некоторые модули могут иметь оба scope: например, список покупок — shared (семейный), а задачи — personal.

#### Поведение по устройствам

| Устройство | Поведение |
|------------|-----------|
| **Телефон / десктоп** | Залогинен как конкретный пользователь. Видит shared + свои personal виджеты |
| **Настенный монитор** | По умолчанию показывает только shared. Тап на аватарку → PIN → показывает shared + personal залогиненного. Через N минут бездействия → автовозврат в shared-режим |

#### Настенный монитор — переключение пользователя

1. В углу экрана — аватарки всех пользователей
2. Тап на аватарку → ввод PIN (4 цифры)
3. Экран дополняется личными виджетами пользователя
4. Таймаут бездействия (настраивается, по умолчанию 5 минут) → возврат в shared
5. Тап на "X" или свайп → ручной выход из личного режима

### Реализация

- Go middleware: проверяет JWT → валидация через JWKS endpoint OIDC провайдера
- Локальные аккаунты: bcrypt хэши в SQLite, выдача JWT при логине
- Сессии: HTTP-only cookie с JWT, refresh token
- PIN: отдельный 4-значный код для быстрого входа на настенном мониторе (хранится как bcrypt хэш)

## Админка / Настройки

Отдельная страница `/admin` (доступна только роли `admin`):

### Общие
- Название дашборда
- Язык, часовой пояс
- Настенный режим: таймаут бездействия, расписание включения/выключения экрана

### Пользователи
- Список пользователей: имя, роль, аватарка, PIN
- Создание / редактирование / удаление
- Назначение ролей

### Внешние ссылки
- CRUD для быстрых ссылок на сервисы (HA, Jellyfin, Immich и т.д.)
- Каждая ссылка: название, URL, иконка, открывать в iframe или новой вкладке
- Отображаются на дашборде как виджет или в боковом меню

### Модули
- Список установленных модулей: название, версия, статус (активен/выключен)
- Включение/выключение модуля
- Кнопка "Перезапустить" (после изменений)

### OIDC
- Подключение провайдера: issuer URL, client ID, client secret
- Маппинг групп → роли
- Тестирование подключения

### Бэкапы
- Автоматический бэкап SQLite по расписанию (ежедневно, хранить N последних)
- Ручной бэкап: кнопка "Скачать бэкап" → скачивает файл SQLite
- Восстановление: загрузка файла → замена БД → перезапуск
- Бэкап хранится в volume `/data/backups/`
