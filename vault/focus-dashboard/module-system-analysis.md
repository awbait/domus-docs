# Анализ модульной архитектуры

Архитектурный разбор модульной системы Focus Dashboard: сильные стороны, ограничения, сравнение с аналогами и план улучшений.

---

## Сильные стороны

### Простота контракта

Интерфейс `Module` — 6 методов. Порог входа для нового builtin-модуля минимальный: достаточно реализовать интерфейс и вызвать `registry.Register()`. Clock-модуль — 20 строк кода.

### Двойная модель: builtin + dynamic

Builtin-модули компилируются в бинарник — максимальная производительность и типобезопасность. Dynamic-модули устанавливаются на лету через ZIP — гибкость для сторонних разработчиков. Оба типа управляются через один Registry.

### Автоматические миграции

`MigrateAll()` запускает миграции только для включённых модулей. Встраивание SQL через `embed.FS` — миграции не теряются и версионируются вместе с кодом.

### Handshake-протокол для dynamic-модулей

Subprocess сообщает о готовности через stdout JSON-строку (`focus-module/1`). ProcessManager ждёт handshake + health check перед проксированием. Это надёжнее, чем фиксированный таймаут.

### Централизованное управление

Registry — единая точка для регистрации, миграций, маршрутизации, health check, включения/выключения. Нет рассыпанной логики.

### Удаление с очисткой layouts

`Delete()` удаляет виджет модуля из всех сохранённых layouts пользователей. Нет "призрачных" виджетов после удаления модуля.

## Ограничения и проблемы

### Общая БД без schema-изоляции

Все модули (builtin и dynamic) работают с одной SQLite БД. Изоляция таблиц — только по конвенции (префикс `pt_` для pill-tracker). Установка модулей — на совести пользователя.

**Перспектива:** При переходе на PostgreSQL решается нативно — отдельная schema per module (`CREATE SCHEMA pill_tracker`) + database user с `GRANT USAGE ON SCHEMA`. В SQLite такой механизм отсутствует.

### Нет sandbox для dynamic-модулей

Subprocess запускается без ограничений ОС. Установка dynamic-модулей — ответственность пользователя, аналогично установке любого стороннего ПО.

### Фронтенд без изоляции

Web Components динамических модулей загружаются как `<script type="module">` в основной DOM. Нет Shadow DOM, нет iframe-sandbox. Аналогично бэкенду — ответственность за установку на пользователе.

### Разный DX для builtin и dynamic

Два runtime — разный опыт разработки. Builtin: Go-интерфейс + React-компонент + запись в `WIDGET_REGISTRY`. Dynamic: manifest.json + бинарник + Web Component. Логическая структура одинаковая, но инструменты и точки входа отличаются. Цель — максимально сблизить подход, чтобы переход между типами был минимальным.

### Нет версионирования API

Маршруты модулей монтируются как `/api/modules/{id}/...` без версии. При обновлении модуля с breaking changes в API нет механизма поддержки старой версии.

### Последовательный health check

`HealthAll()` проверяет модули последовательно. При большом количестве модулей или медленном health check одного модуля — задержка для всех.

### Нет dependency injection

Модули получают `*sql.DB` напрямую. Нет абстракции для общих сервисов ядра (push-уведомления, WebSocket, настройки). Каждый модуль должен сам знать про API ядра.

### Нет event system

Модули не могут подписываться на события друг друга. Пример: pill-tracker не может уведомить domovoy-control о пропущенной дозе без прямой зависимости.

## План улучшений

### Фаза 1 — Унификация DX

**Проблема:** Разный подход к разработке builtin и dynamic модулей при одинаковой логической структуре.

**Решение:**
- Один модуль может предоставлять несколько виджетов. Widget ID — составной: `{moduleId}.{widgetId}`
- Каждый виджет описывает размеры: `min` (1x1), `max` (4x3), `default` (2x2)
- Расширить `Describer` / `Meta` массивом `Widgets` с метаданными (id, name, sizes, icon) — то, что dynamic-модули описывают в `manifest.json`
- Фронтенд получает полный список виджетов из `GET /api/modules` вместо хардкода в `WIDGET_REGISTRY`
- Builtin React-компоненты подгружаются через `React.lazy()` по widget ID
- Единая документация: один гайд "как написать модуль" с пометками где отличается runtime

**Трудоёмкость:** Низкая-средняя. Расширить `ModuleInfo`, рефакторить `widgets.tsx`, обновить `GET /api/modules`.

### Фаза 2 — Widget Bridge SDK

**Проблема:** Dynamic-виджеты делают `fetch()` напрямую, без стандартизированного API. Каждый виджет сам разбирается с URL-ами, настройками, событиями.

**Решение (вдохновлено Homey Widget SDK):** JavaScript-библиотека `focus-widget-sdk`, которую подключает каждый dynamic-виджет:

```js
import { Focus } from 'focus-widget-sdk'

Focus.ready()                          // виджет загружен
Focus.api('GET', '/data')              // запрос к своему бэкенду (проксируется)
Focus.getSettings()                    // настройки экземпляра виджета
Focus.on('settings:changed', cb)       // подписка на изменение настроек
Focus.getWidgetId()                    // ID экземпляра
```

SDK инкапсулирует URL-маршрутизацию, auth-токены, WebSocket-подписки. Одинаковый API для builtin и dynamic.

**Трудоёмкость:** Средняя. Написать SDK + адаптировать существующие виджеты.

### Фаза 3 — Widget Preview

**Проблема:** В Widget Picker только текст и иконка — непонятно как виджет выглядит до добавления.

**Решение (вдохновлено Homey):** Каждый модуль предоставляет `preview.png` — скриншот виджета. Widget Picker показывает превью при выборе.

- Builtin: изображения через `embed.FS`, отдаются по `GET /api/modules/{id}/preview.png`
- Dynamic: `preview.png` в ZIP-архиве

**Трудоёмкость:** Низкая. Добавить endpoint + обновить Widget Picker UI.

### Фаза 4 — Core Services API

**Проблема:** Модули не имеют доступа к сервисам ядра (push, WebSocket, settings).

**Решение:** Набор HTTP-эндпоинтов, доступных модулям:
- `POST /internal/push` — отправить push-уведомление
- `POST /internal/ws/broadcast` — отправить WebSocket-сообщение
- `GET /internal/settings/{key}` — прочитать глобальную настройку
- `POST /internal/events` — опубликовать событие

Для builtin-модулей — Go-интерфейс `CoreServices`, передаваемый при создании модуля.

**Трудоёмкость:** Средняя. Проектирование API + реализация для обоих типов модулей.

### Фаза 5 — Event Bus

**Проблема:** Модули не могут взаимодействовать между собой.

**Решение:** Pub/Sub шина событий в ядре. Модули подписываются на типы событий, ядро маршрутизирует. Для builtin — Go channels, для dynamic — WebSocket или HTTP webhooks.

```go
type Event struct {
    Type      string          // "pill-tracker.dose.missed"
    Source    string          // "pill-tracker"
    Payload   json.RawMessage
    Timestamp time.Time
}
```

**Трудоёмкость:** Средняя. Основной дизайн простой, но нужно продумать гарантии доставки и ordering.

### Шаблонный модуль

Когда DX унифицирован — создать `example-counter` в `focus-modules/`: минимальный fullstack модуль (Go backend + Web Component) как образец для сторонних разработчиков.

### На будущее (при наличии публичного marketplace)

- **Schema-изоляция данных** — при переходе на PostgreSQL: отдельная schema per module + database user с ограниченными правами
- **Sandbox для subprocess** — `unshare` (network namespace) + seccomp-профиль, AppArmor на production
- **Frontend isolation** — iframe с `sandbox="allow-scripts"` + `postMessage` bridge

Эти меры актуальны только при установке недоверенных модулей из публичного каталога. Пока модули устанавливаются осознанно — ответственность на пользователе.

| Фаза | Приоритет | Обоснование |
