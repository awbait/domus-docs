# domovoy-docs

Obsidian vault с документацией проекта Домовой.

---

## Файлы и их назначение

### Продукт и задачи
| Файл | Назначение | Обновлять когда |
|------|-----------|-----------------|
| `project.md` | Обзор для потребителя: текущие возможности, запланированное, стек | При выполнении задач из Planned/Backlog |
| `tasks.md` | Kanban-доска: статус задач, прогресс | При старте/завершении задачи |
| `implementation-plan.md` | Полное описание каждой задачи: архитектура, код, зависимости | При добавлении новой задачи или уточнении существующей |

### Архитектура (`arch/`)
| Файл | Назначение | Обновлять когда |
|------|-----------|-----------------|
| `arch/overview.md` | Схема системы, компоненты, протоколы, deployment | При добавлении/удалении сервисов |
| `arch/pipeline.md` | Голосовой пайплайн, режимы работы, latency | При изменении шагов пайплайна |
| `arch/services.md` | ML-сервисы: модели, конфиги, gRPC контракты | При смене моделей или контрактов |
| `arch/security.md` | Безопасность: текущее состояние и планы | При реализации security-задач |
| `arch/hub-spoke.md` | Hub-Spoke архитектура (планируется) | При реализации P3-задач |

### Разработка
| Файл | Назначение | Обновлять когда |
|------|-----------|-----------------|
| `go-practices.md` | Архитектурные требования и Go best practices | При принятии новых архитектурных решений |

## Правило: задачи дублируются в двух местах

**`implementation-plan.md`** — источник истины. Полное описание: что делать, какие файлы трогать, примеры кода, зависимости.

**`tasks.md`** — Kanban-доска. Краткое название + ссылка на хедер в `implementation-plan.md`:

```
- [ ] Название задачи #тег #приоритет
	Одна строка описания. [[implementation-plan#Название задачи]]
```

Ссылка в Obsidian: `[[implementation-plan#Название задачи]]` — открывает нужный раздел напрямую.

## Теги задач

### Функциональные (что это)
| Тег | Назначение |
|-----|-----------|
| `#pipeline` | Голосовой пайплайн (orchestrator, STT, TTS, LLM) |
| `#security` | Безопасность (mTLS, шифрование, секреты, rate limiting) |
| `#infra` | Инфраструктура (Docker, деплой, OTA, hub-spoke) |
| `#arch` | Архитектура (рефакторинг, паттерны, структура пакетов) |
| `#ml` | ML-модели (wake word, speaker recognition, LLM routing) |
| `#ui` | Интерфейсы (web dashboard) |

### Приоритет (когда делать)
| Тег | Условие |
|-----|---------|
| `#p1` | Без блокеров, делать сейчас |
| `#p2` | После P1 |
| `#p3` | После P2 (требует hub-spoke и mTLS) |
| `#backlog` | Без конкретных сроков |
